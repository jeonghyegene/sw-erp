# 인사·근태 문서 5분류 구조 근거

> 문서 상태: source-only 재정리 v0.2
> 기준일: 2026-07-16
> 목적: `docs/hr-attendance` 산출물을 `entities / permissions / policies / screens / states` 5개 폴더로 나눈 것이 합리적인지 평가하고, 그 근거·경계·개선 방향을 한곳에 정리한다.
> 성격: 폴더 분류의 설계 근거 문서. 개별 산출물의 내용은 각 폴더와 `AGENTS.md`를 진실 공급원으로 한다.

---

## 1. 결론 요약

**합리적이다.** 5개 폴더는 "무엇을 담느냐"가 아니라 **개발에 필요한 업무 명세를 서로 겹치지 않는 관점 축(관심사)으로 나눈 것**이며, 각 폴더는 그 관점의 **단일 진실 공급원(SoT, Single Source of Truth)** 역할을 한다. 이 경계는 이미 `AGENTS.md §7.1`에 규약으로 명문화되어 있으므로, 5개 폴더는 즉흥적 산물이 아니라 **규약의 구현**이다.

한 가지만 분명히 해둘 점: `AGENTS.md`가 정의한 표준 구조는 원래 **8개 폴더**(`screens / entities / policies / permissions / states` + `processes / exceptions / templates`)다. 현재는 그중 **5개만 실제 생성**된 상태이고, 나머지 3개는 아직 별도 폴더로 필요할 만큼 내용이 쌓이지 않아 다른 문서에 흡수되어 있다(§4.2). 따라서 "5개로 정의했다"는 표현보다 **"표준 8분류 중, 지금 필요한 5개를 실체화했다"**가 정확하다. 이 점을 근거로 삼으면 방어가 훨씬 견고해진다.

| 항목 | 판단 |
|---|---|
| 5개 분류의 논리적 타당성 | 타당 — 관심사 분리 + SoT 원칙 충족 |
| 상호배타성(중복 없음) | 대체로 충족 — 경계 겹침 3곳은 `AGENTS.md` 규칙으로 해소됨 (§4.1) |
| 전체포괄성(누락 없음) | 조건부 충족 — 업무흐름·예외·용어 축은 현재 다른 문서에 흡수, 볼륨 임계 시 승격 필요 (§4.2) |
| 프로젝트 규약과의 정합 | 정합 — `AGENTS.md §7.1` SoT 경계와 1:1 일치 |

## 2. 5개 폴더가 각각 답하는 질문

분류의 본질은 "각 폴더가 서로 다른 하나의 질문에만 답한다"는 것이다. 이것이 성립하면 분류는 타당하다.

| 폴더 | 답하는 질문 | 관리 키 | 파일 수 | 이 폴더가 SoT인 대상 |
|---|---|---|---:|---|
| **screens/** | 이 화면에서 **무엇을 할 수 있나** (액션·처리·결과) | 화면 ID | 26 | 화면별 업무 액션·입력·조건·처리·결과·예외 |
| **permissions/** | 그 액션을 **누가** 할 수 있나 | 화면 ID | 27 | 액션별 실행 주체·데이터 범위·필드 제한 |
| **entities/** | 그 업무가 다루는 **데이터가 무엇인가** | 도메인 | 18 | 업무 엔티티의 의미·관계·불변 규칙 |
| **states/** | 그 데이터가 **어떤 생명주기**를 갖나 | 엔티티 | 29 | 엔티티 상태값과 상태 전이 |
| **policies/** | 여러 화면에 걸친 **공통 규칙이 무엇인가** | 정책 주제 | 27 | 재사용되는 처리 정책·제약 |

각 폴더가 답하는 질문이 서로 다르다(무엇을 한다 / 누가 / 데이터가 뭐다 / 어떻게 변한다 / 규칙이 뭐다). 질문이 겹치지 않으므로 분류 축이 유효하다.

## 3. 왜 이 5분류가 맞는가 — 근거

### 3.1 관심사 분리 (Separation of Concerns)

한 문서가 하나의 관심사만 책임지면, 같은 사실이 여러 곳에 복제되지 않고 수정 지점이 하나로 고정된다. 예를 들어 "고용계약이 서명완료로 바뀌는 조건"은 `states/EMPLOYMENT_CONTRACT_STATE.md` 한 곳에만 있고, 화면 문서는 이를 참조만 한다. 관심사가 섞인 단일 문서였다면 화면마다 상태·권한·규칙이 재서술되어 불일치가 생긴다. 5분류는 이 복제를 구조적으로 차단한다.

### 3.2 기획 UX 아키텍처 축과 정합

이 프로젝트의 기획 관점 축(목표 → 사람 → 업무흐름 → 데이터 → 기능/UI → 정책/규칙 → 연계 → 운영)에 5개 폴더가 정확히 대응한다.

| UX 아키텍처 축 | 대응 폴더 | 비고 |
|---|---|---|
| 사람 (Who) | **permissions/** | 액션별 실행 주체·범위 |
| 업무흐름 (How) | **screens/** (진입점) + **states/** (시간축) | 흐름의 화면 순서는 루트 `01_PROCESS_MAP.md` |
| 데이터 (What) | **entities/** | 업무 엔티티 |
| 기능/UI (Interface) | **screens/** | 화면 액션 |
| 정책/규칙 (Rule) | **policies/** | 공통 규칙 |
| 목표 / 연계 / 운영 | (별도 폴더 없음) | 목표는 화면 목적, 연계는 화면 관계로 흡수 (§4.2) |

즉 5개 폴더는 기획 축의 핵심 5개(Who·What·Interface·Rule·생명주기)를 실체화한 것이다.

### 3.3 소프트웨어 표준 산출물과 1:1 대응

5분류는 도메인 주도 설계(DDD)와 UML의 표준 산출물 구분을 그대로 따른다. 임의 분류가 아니라 업계 표준의 매핑이다.

| 폴더 | 대응 표준 산출물 | 개발 시 소비처 |
|---|---|---|
| entities/ | 도메인 모델 / Aggregate·Entity (→ ERD 입력) | 백엔드 DB·ERD 설계 |
| states/ | 상태 기계 / UML Statechart | 백엔드 상태 전이·검증 |
| policies/ | 비즈니스 규칙 / 불변식(Invariant) | 백엔드 검증 로직 |
| permissions/ | 인가 모델 / RBAC 매트릭스 | 백·프론트 접근 제어 |
| screens/ | 유스케이스 / 기능·UI 명세 | 프론트 화면 구현 |

### 3.4 개발 인계 관점 — 백엔드·프론트가 무엇을 어디서 읽는가

`AGENTS.md §3.4`는 이 산출물을 "기획자가 검증하고 개발자가 ERD·DB·API를 설계할 때 쓰는 업무 입력자료"로 규정한다. 5분류는 개발자가 필요한 정보를 곧장 찾게 한다.

| 개발 작업 | 참조 폴더 |
|---|---|
| 테이블·컬럼·관계 설계 | entities/ |
| 상태 컬럼·전이 검증 | states/ |
| 서버 검증·업무 규칙 | policies/ + screens/ (화면 국한 규칙) |
| API 인가·데이터 범위 | permissions/ |
| 화면 구현·버튼·필드·흐름 | screens/ |
| 상태 뱃지·버튼 노출 제어(프론트) | states/ + permissions/ |

### 3.5 변경 파급의 국소화

권한 정책만 바뀌면 `permissions/`만, 상태 하나 추가되면 `states/`의 해당 엔티티 문서만 고치면 된다. 분류가 없거나 화면 단위로만 묶였다면 한 규칙 변경이 26개 화면 문서로 번진다. 5분류는 변경 영향 범위를 폴더 단위로 가둔다.

### 3.6 프로젝트 규약이 이미 이 경계를 명문화

가장 강한 근거다. `AGENTS.md §7.1`은 각 산출물의 진실 공급원을 다음과 같이 못박아 두었다.

- 액션 목록·업무 처리 → `screens/{SCREEN_ID}_ACTIONS.md`
- 실행 권한·범위·마스킹 → `permissions/{SCREEN_ID}_PERMISSION_MATRIX.md`
- 엔티티 의미·관계·불변 규칙 → `entities/{DOMAIN}_ENTITIES.md`
- 공통 처리 정책 → `policies/{TOPIC}_POLICY.md`
- 상태와 상태 전이 → `states/{ENTITY}_STATE.md`

5개 폴더는 이 규약의 다섯 SoT를 그대로 폴더로 실체화한 것이다. 분류 근거를 "왜 5개인가"로 물으면 답은 "규약이 정의한 SoT가 다섯 개이기 때문"이다.

### 3.7 두 개의 조직 키를 의도적으로 병용

5개 폴더는 파일 명명 규칙이 두 종류로 갈린다. 이것은 실수가 아니라 관점의 차이를 반영한 설계다.

| 조직 키 | 폴더 | 이유 |
|---|---|---|
| **화면 축** (화면 ID로 명명) | screens/, permissions/ | 화면과 1:1. 화면 하나를 구현·검수할 때 두 파일만 열면 됨 |
| **도메인 축** (개념/엔티티/주제로 명명) | entities/, states/, policies/ | 여러 화면이 공유하는 개념. 화면마다 재정의하면 불일치 발생 |

화면에 종속적인 것(액션·권한)은 화면 키로, 화면을 가로지르는 것(데이터·상태·규칙)은 도메인 키로 나눈 것이다. `AGENTS.md`가 "상태 코드를 화면별로 중복 정의하지 않는다(§14)"고 금지한 것과 같은 원리다.

## 4. 비판적 검토 (MECE)

### 4.1 상호배타성 — 경계가 겹쳐 보이는 3곳

완전히 직교하지는 않는다. 겹쳐 보이는 지점이 세 곳 있으나, 모두 `AGENTS.md`에 해소 규칙이 존재한다.

| 겹침 지점 | 왜 겹쳐 보이나 | 해소 규칙 | 상태 |
|---|---|---|---|
| **states ↔ policies** | 상태 전이 조건도 일종의 "규칙"이라 policies로 볼 여지 | `AGENTS.md §11`: 상태 변경 조건은 `states/`, 여러 화면 공통 규칙은 `policies/` | 규칙으로 관리 가능 |
| **permissions ↔ policies** | `00_PERMISSION_POLICY.md`는 성격상 "정책"인데 `permissions/`에 위치 | `AGENTS.md §11`: 권한 조건은 `permissions/`로 분리 | 의도된 배치 |
| **screens ↔ entities/states/permissions** | 화면 문서가 액션마다 Actor·참조 엔티티·예외를 언급 → 중복처럼 보임 | 화면 문서는 "그 화면에 국한된 것"만, 공통은 ID·경로로 참조만 | SoT 원칙으로 해소 |

판정: 경계가 겹치는 지점은 있으나 **판단 규칙이 문서화되어 있어 운영상 모호하지 않다.** 다만 이 규칙을 모르는 신규 작성자는 상태 전이 조건을 policies에 잘못 넣을 수 있으므로, §5의 개선안 3을 권한다.

### 4.2 전체포괄성 — 개발에 필요한데 별도 폴더가 없는 축

개발 명세에 필요한 축 대비 현재 커버리지를 점검하면, 세 축이 별도 폴더 없이 다른 문서에 흡수되어 있다.

| 필요한 축 | 현재 위치 | 별도 폴더(`AGENTS.md` 예약) | 승격 필요 시점 |
|---|---|---|---|
| 업무흐름 (화면 선후행·프로세스) | 루트 `01_PROCESS_MAP.md` 단일 파일 + 화면 문서의 "선행·연결 화면" | `processes/` | 프로세스별 상세 흐름이 단일 맵에 담기 복잡해질 때 |
| 공통 예외·복구 | 각 화면 문서의 "예외" 항목 | `exceptions/` | 실패·복구 패턴이 여러 화면에 반복될 때 |
| 작성 서식 | (없음) | `templates/` | 산출물 서식을 표준화·재사용할 때 |
| 용어집 (Ubiquitous Language) | entities에 부분 흡수 | (규약에 없음) | 용어 불일치가 문서 간 발생할 때 |
| 외부 연계 (System) | policies/screens에 흡수 | (규약에 없음) | 외부 시스템 연동이 실제로 정의될 때 |
| API 계약 | **의도적 제외** | — | `AGENTS.md §3.4` 기획-개발 경계상 제외가 정당 |

판정: **API 계약 제외는 정당**하다(기획 산출물 경계). 업무흐름·예외·서식은 `AGENTS.md`가 이미 폴더로 예약해 두었으므로 "누락"이 아니라 "미실체화"다. 현재 볼륨에서는 흡수가 합리적이고, 임계를 넘으면 승격하면 된다. 즉 5분류는 닫힌 체계가 아니라 **8분류 표준의 점진적 실체화 단계**로 보는 것이 정확하다.

## 5. 개선안

| 우선순위 | 개선안 | 근거 |
|---|---|---|
| 1 | **이 근거 문서(본 파일)를 색인에 등록**하고 폴더 구조의 공식 설명으로 유지 | 분류 의도가 문서화되지 않으면 신규 작성자가 규칙을 재해석 |
| 2 | `00_DOCUMENT_INDEX.md`의 "문서별 확인 기준" 표에 **각 폴더=SoT 매핑**을 명시 (§2 표 반영) | 폴더의 역할을 색인에서 바로 확인 |
| 3 | screens·states·policies 경계의 **판단 규칙 1줄 요약**을 각 폴더 첫 문서 상단 또는 본 문서에 상시 노출 | §4.1 겹침 지점의 오배치 예방 |
| 4 | 업무흐름/예외 내용이 임계(예: 화면 문서 예외가 5개 화면 이상 반복)를 넘으면 **`processes/` · `exceptions/` 폴더 승격** | `AGENTS.md §7` 표준 구조 복원 |
| 5 | 용어 불일치가 관찰되면 **`GLOSSARY.md` 1개 추가** 검토 | DDD 유비쿼터스 언어; 문서 간 용어 일치는 `AGENTS.md §8` 정합성 검사 대상이기도 함 |

개선안 1~3은 지금 바로 반영 가능한 문서 작업이고, 4~5는 볼륨이 커질 때의 조건부 확장이다.

## 6. 한 줄 결론

5개 폴더는 **관심사 분리와 단일 진실 공급원 원칙에 따른 타당한 분류**이며, `AGENTS.md §7.1`이 정의한 표준 8분류 중 현재 필요한 다섯을 실체화한 것이다. 중복은 규칙으로 통제되고, 누락으로 보이는 축은 폴더로 예약되어 있어 확장 여지가 열려 있다.
