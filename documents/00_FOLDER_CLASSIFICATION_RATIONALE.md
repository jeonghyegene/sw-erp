# 인사·근태 문서 5분류 근거

> 문서 상태: source-only 재정리 v0.2
> 기준일: 2026-07-16
> 목적: `docs/hr-attendance` 산출물을 `entities / permissions / policies / screens / states` 5개 폴더로 나눈 것의 분류 본질과 근거를 정리한다.

---

## 1. 분류의 본질

5개 폴더의 본질은 **각 폴더가 서로 다른 하나의 질문에만 답한다**는 것이다. 질문이 겹치지 않으므로 분류 축이 유효하다. 순서는 **데이터 실체(안쪽) → 사용자 표층(바깥쪽)** 의 의존 방향을 따른다. 각 단계는 앞 단계가 정의돼야 성립한다.

| 순서 | 폴더 | 답하는 질문 | 관리 키 |
|---|---|---|---|
| 1 | **entities/** | 그 업무가 다루는 **데이터가 무엇인가** | 도메인 |
| 2 | **states/** | 그 데이터가 **어떤 생명주기**를 갖나 (엔티티에 종속된 내재 속성) | 엔티티 |
| 3 | **policies/** | 여러 엔티티·상태를 가로지르는 **공통 규칙이 무엇인가** | 정책 주제 |
| 4 | **permissions/** | 그 액션을 **누가** 할 수 있나 (데이터·규칙 위의 접근 통제층) | 화면 ID |
| 5 | **screens/** | 이 화면에서 **무엇을 할 수 있나** (모든 것을 묶은 표층) | 화면 ID |

데이터가 뭐다 → 어떻게 변한다 → 규칙이 뭐다 → 누가 → 무엇을 한다. 좁은 관심사(엔티티 1개에 갇힌 states)가 넓은 관심사(여러 엔티티를 가로지르는 policies)보다 앞에 온다.

---

## 2. 5분류의 근거

### 2.1 관심사 분리

한 문서가 하나의 관심사만 책임지면, 같은 사실이 여러 곳에 복제되지 않고 수정 지점이 하나로 고정된다. "고용계약이 서명완료로 바뀌는 조건"은 `states/EMPLOYMENT_CONTRACT_STATE.md` 한 곳에만 두고 화면 문서는 참조만 한다. 관심사가 섞였다면 화면마다 상태·권한·규칙이 재서술되어 불일치가 생긴다. 5분류는 이 복제를 구조적으로 차단한다.

### 2.2 기획 관점과 정합

기획 관점 축(목표 → 사람 → 업무흐름 → 데이터 → 기능/UI → 정책/규칙 → 연계 → 운영)의 핵심에 5개 폴더가 대응한다.

| 기획 축 | 대응 폴더 |
|---|---|
| 사람 (Who) | permissions/ |
| 데이터 (What) | entities/ |
| 기능/UI (Interface) | screens/ |
| 정책/규칙 (Rule) | policies/ |
| 업무흐름·생명주기 (How) | states/ (+ 화면 순서는 `01_PROCESS_MAP.md`) |

### 2.3 소프트웨어 표준 산출물과 1:1 대응

임의 분류가 아니라 개발 산출물의 표준 구분을 따른다.

| 폴더 | 대응 표준 산출물 | 개발 시 소비처 |
|---|---|---|
| entities/ | 도메인 모델 (→ ERD 입력) | 백엔드 DB·ERD 설계 |
| states/ | 상태 기계 (상태·전이 정의) | 백엔드 상태 전이·검증 |
| policies/ | 업무 규칙 | 백엔드 검증 로직 |
| permissions/ | 접근 권한 정의 | 백·프론트 접근 제어 |
| screens/ | 기능·UI 명세 | 프론트 화면 구현 |

### 2.4 개발 인계

개발자가 필요한 정보를 곧장 찾을 수 있게 한다.

| 개발 작업 | 참조 폴더 |
|---|---|
| 테이블·컬럼·관계 설계 | entities/ |
| 상태 컬럼·전이 검증 | states/ |
| 서버 검증·업무 규칙 | policies/ + screens/ |
| API 인가·데이터 범위 | permissions/ |
| 화면 구현·버튼·필드·흐름 | screens/ |

### 2.5 두 개의 조직 키

파일 명명 규칙이 두 종류로 갈리며, 이는 관점 차이를 반영한 설계다.

| 조직 키 | 폴더 | 이유 |
|---|---|---|
| **화면 축** (화면 ID로 명명) | screens/, permissions/ | 화면과 1:1. 화면 하나를 구현·검수할 때 두 파일만 열면 됨 |
| **도메인 축** (개념·주제로 명명) | entities/, states/, policies/ | 여러 화면이 공유하는 개념. 화면마다 재정의하면 불일치 발생 |

화면에 종속적인 것(액션·권한)은 화면 키로, 화면을 가로지르는 것(데이터·상태·규칙)은 도메인 키로 나눈다.

## 3. 필요 보완점

현재 5분류로 담기지 않고 다른 문서에 흡수되어 있는 두 축은, 내용이 커지면 별도 관리가 필요하다.

| 보완 축 | 현재 위치 | 보완 필요 시점 |
|---|---|---|
| **업무흐름** (화면 선후행·프로세스) | 루트 `01_PROCESS_MAP.md` 단일 파일 + 화면 문서의 "선행·연결 화면" | 프로세스별 상세 흐름이 단일 맵에 담기 복잡해질 때 별도 관리 |
| **외부 연동** (System 연계) | policies/·screens/에 부분 흡수 | 외부 시스템 연동이 실제로 정의될 때 별도 관리 |
