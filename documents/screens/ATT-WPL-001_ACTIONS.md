## WS-A03 근무조 추가

- 화면: ATT-WPL-001 근무조 설정 탭
- 화면 위치: 툴바 `[+ 근무조 추가]`(`data-shift-act="add"`) → 추가 모달(`modal-shift-editor`)
- 실행 주체: 인사/근태 담당
- 필요 권한: Create (WS-A03)
- 현행 CRUD 권한 매핑: Create
- 데이터 접근 범위: 전사
- 필드 노출·마스킹: 없음
- 입력: 근무조 명(label), 출·퇴근 시각, 휴게 1·2구간, 색상
- 참조 데이터: 부서 목록(사용 부서 선택용), 색상 팔레트
- 선행 조건: 없음(근무조는 선행 최상위 마스터)
- 처리: 산정값 자동 계산(`recompute`) → 코드 자동 채번(`WT`+D/N+일련번호) → 마스터에 추가
- 생성·변경 데이터: 신규 근무조 레코드(활성·미사용)
- 출력·처리 결과: 목록에 신규 행 추가, 완료 토스트(좌하단)
- 후행 사용처: 부서 관리 기본 근무조 선택지, 근무스케줄 편성
- 상태 변경: (없음) → 활성·미사용
- 성공 시 사용자 알림: 토스트 "근무조가 추가되었습니다"(결과 알림)
- 검증 실패: 시각 형식·필수 미입력·휴게 정책 위반 → **인라인 필드 오류**(토스트 금지) (`validateBasics` `:913`, `validateBreakPolicy` `:924`)
- 업무 예외: 동일 산정 기준 중복 등록 여부는 확인 대상
- 시스템 실패 시 처리: `미결정`(mock)
- 취소 처리: 모달 취소 시 미저장
- 재처리 가능 여부: 가능(다시 추가)
- 근거 소스: `page-att-settings.js:328`, `att-shift-data.js:565`(renderAddModal)·`:952`(saveAdd)
- 확정 상태: `확정`(추가는 결재 없이 자유)
- 미결정 사항: 추가에 승인 통제가 필요한지

## WS-A04 근무조 수정

- 화면: ATT-WPL-001 근무조 설정 탭
- 화면 위치: 행 상세/수정 진입 → 인-페이지 수정 화면(host에 `editInto` 마운트)
- 실행 주체: 담당
- 필요 권한: Update (WS-A04)
- 현행 CRUD 권한 매핑: Update
- 데이터 접근 범위: 전사
- 필드 노출·마스킹: **사용 이력 있으면 산정 기준 필드 잠금**(POL-WS-02)
- 입력: 명칭·출퇴근·휴게·색상. 부서 연결 근무조는 **적용 시작일·사유 필수**
- 참조 데이터: 기존 근무조 값, 부서 연결 현황
- 선행 조건: `canEdit` = 사용 부서 0 + 사용 이력 없음 + 기본 아님 + 활성 + 대기 아님
- 처리: 변경 필드 diff 산출 → 산정값 재계산 → 수정 이력 기록. 적용 시작일 지정 시 예약(`promotePending`)
- 생성·변경 데이터: 근무조 레코드 갱신, `shiftHistory` 이력
- 출력·처리 결과: 목록 갱신, 완료 토스트
- 후행 사용처: 변경된 산정 기준이 이후 배치·근태에 반영
- 상태 변경: 없음(속성 변경)
- 성공 시 사용자 알림: 토스트 "수정되었습니다"
- 검증 실패: 적용 시작일·사유 미입력, 시각 오류 → 인라인 오류(`editErrors` `:244`)
- 업무 예외: 사용 이력/사용 중/기본 근무조는 진입 자체 차단 + 잠금 안내
- 시스템 실패 시 처리: `미결정`
- 취소 처리: `[뒤로]`로 그리드 복귀, 미저장
- 재처리 가능 여부: 가능
- 근거 소스: `att-shift-data.js:598`(renderEditInto)·`:363`(diffShift)·`:242`(editMeta)
- 확정 상태: `확정`(잠금·이력) / 적용 시작일의 근태 소급 영향 `구현 갭`
- 미결정 사항: 색상만 수정을 별도 권한으로 분리할지(WS-A12)

## WS-A05 근무조 미사용(비활성화) 상신

- 화면: ATT-WPL-001 근무조 설정 탭
- 화면 위치: 행 액션 → 미사용 처리 → 전자결재 상신 모달(`openCodeApprovalModal('비활성')`)
- 실행 주체: 담당(상신) / 승인자(승인)
- 필요 권한: 현행상 Update (WS-A05) — **권한 갭**(상신 권한 분리 검토)
- 현행 CRUD 권한 매핑: Update(상신) + 결재 승인 권한(별도)
- 데이터 접근 범위: 전사
- 필드 노출·마스킹: 없음
- 입력: 변경 사유(필수)
- 참조 데이터: 근무조 상태 플래그(`codeFlags`)
- 선행 조건: `canDeactivate` = 사용 부서 0 + 활성 + 기본 아님 + 대기 아님
- 처리: `pendingChange` 설정(승인대기) → 승인 시 `active=false` + `usageLog`·`codeChangeLog` 기록
- 생성·변경 데이터: 상태 → 승인대기 → 비활성. 변경 이력·사용 이력
- 출력·처리 결과: 승인대기 뱃지, 승인 후 비활성 마커
- 후행 사용처: 비활성 근무조는 배치 후보에서 제외
- 상태 변경: 활성·미사용 → 승인대기 → 비활성 (→ [상태 문서](../states/WORK_SHIFT_STATE.md))
- 성공 시 사용자 알림: 상신 접수/승인 완료 토스트
- 검증 실패: 사유 미입력 → 인라인 오류(`codeAct.error`)
- 업무 예외: 사용 중·기본·이미 대기 → 액션 차단 + 잠금 안내
- 시스템 실패 시 처리: `미결정`(전자결재 연계 외부)
- 취소 처리: 승인 전 상신 취소(대기 해제) — `추정`(취소 UI 확인 대상)
- 재처리 가능 여부: 반려 시 재상신 가능 — `추정`
- 근거 소스: `att-shift-data.js:977`(submitCodeChange)·`:987`(approveCodeChange)·`:1067`(openCodeApprovalModal)
- 확정 상태: `확정`(상신→승인 흐름) / 승인 트랜잭션 백엔드 `미결정`
- 미결정 사항: 상신 권한 분리, 반려·재상신 절차

## WS-A06 근무조 재개(활성화)

- 화면 위치: 비활성 근무조 행 액션 → 재개
- 실행 주체: 담당
- 필요 권한: 현행상 Update (WS-A06) — **권한 갭**

## WS-A07 근무조 삭제 상신

- 화면 위치: 행 액션 → 삭제 → 전자결재 상신
- 실행 주체: 담당(상신) / 승인자(승인)
- 필요 권한: Delete(상신) + 결재 승인 권한(별도)
- 선행 조건: `canDelete` = 사용 부서 0 + 사용 이력 없음 + 기본 아님 + 활성 + 대기 아님
- 처리: `pendingChange('삭제')` → 승인 시 목록에서 제거, `codeChangeLog` 존속
- 상태 변경: 활성·미사용 → 승인대기 → 삭제됨
- 검증 실패: 사유 미입력 → 인라인 오류
- 업무 예외: 사용 이력 있으면 삭제 대신 미사용만 허용
- 취소 처리: 승인 전 취소 / 삭제 승인 후 복구 불가(재등록)
- 근거 소스: `att-shift-data.js:977`(submitCodeChange '삭제')·`:975`(caTypeWord)
- 확정 상태: `확정`(조건·흐름) / 물리·논리 삭제 `미결정`
- 미결정 사항: 삭제 후 이력 보존 기간
