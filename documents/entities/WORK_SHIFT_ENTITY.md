## 1. 엔티티 개요

| 항목 | 내용 | 상태 |
|---|---|---|
| 엔티티명 | 근무조 (WORK_SHIFT) | 확정 |
| 업무 정의 | 출·퇴근 시각, 총 근무시간, 연장·심야시간, 휴게(간식) 등 **근태 산정의 기준 단위**. 직원·부서에 배정되어 근무스케줄 편성과 근태 정상/이상 판정, 초과수당·식권 산정의 기준이 된다. | 확정 |
| 진실 공급원(SoT) | `App.AttShifts`(`att-shift-data.js` `STATE.shifts`). 근태현황·근무스케줄 편성·근로계약·식권 화면이 모두 이 마스터를 읽어 매핑한다. | 확정 (`att-shift-data.js:1~13` 모듈 주석, 공개 API) |
| 소유 화면 | `ATT-WPL-001` 근무조 설정 탭 (등록·수정·상태 변경 입력점). 편집 로직은 `App.AttShifts` 모듈을 재사용. | 확정 (`page-att-settings.js:5`) |

근무조는 고유 식별자·독립적 생성/변경 주기·상태(활성·미사용)·사용 이력·다른 데이터(부서·직원)와의 필수 관계를 모두 가지므로 독립 엔티티로 판정한다(프로젝트 §13 엔티티 후보 기준 충족).

---

## 2. 고유 식별자

| 식별자 | 규칙 | 상태 |
|---|---|---|
| `code` | 시스템 자동 채번. `WT` + 근무시간유형(`D`=주간 / `N`=야간) + 일련번호 2자리. 예: `WTD01`, `WTN02`. 심야(22:00~06:00) 겹침 여부로 D/N 자동 분류. | 확정 (`att-shift-data.js:172~176`, `channelCode()` `:317`) |

- `code`는 사용자 입력이 아니라 등록 시 시스템이 부여한다.
- `label`(근무조 명, 예: A조·B조)은 `code`와 **별개 namespace**이며, 배치·현황 등 타 화면은 `label`로 표기한다. → `label`은 식별자가 아니라 표시명. (확정 — `att-shift-data.js:177` 주석)

## 3. 주요 속성

| 속성 | 설명 | 산출 방식 | 상태 |
|---|---|---|---|
| `code` | 근무조 코드(식별자) | 자동 채번 | 확정 |
| `label` | 근무조 명(표시명) | 사용자 입력 | 확정 |
| `useDepts` | 사용 부서 목록(시드) | 입력 + 파생 | 확정 |
| `start` / `end` | 출근 / 퇴근 시각(HH:MM). 퇴근 ≤ 출근이면 야간조(익일) | 사용자 입력 | 확정 (`:43~50`) |
| `breakStart`/`breakEnd`, `breakStart2`/`breakEnd2` | 휴게(간식) 1·2구간 | 사용자 입력 | 확정 |
| `workHours` | 총 근무시간(출~퇴 − 휴게), 'H:MM' | `recompute()` 자동 | 확정 (`:80~92`) |
| `breakMin` | 휴게 총 분 | 자동 | 확정 |
| `otMin` | 연장시간 분 = 실근로 − 8시간(소정근로) 초과분 | 자동 | 확정 |
| `nightMin` | 심야시간 분 = 근무구간과 22:00~06:00 겹침 | 자동 | 확정 |
| `isNight` | 심야 포함 시 야간 근무조 자동 판정 (→ 코드 D/N 결정) | 자동 | 확정 |
| `color` | 근무조 색상(기본 회색 + 7색 팔레트). 스케줄 시각 구분용 | 사용자 지정 | 확정 (`:200~209`) |
| `active` | 활성/비활성(별도 축). 비활성 = 배치 후보에서 제외(폐기) | 상태 변경 | 확정 (→ [WORK_SHIFT 상태](../states/WORK_SHIFT_STATE.md)) |
| `startedAt` | 사용 시작일(등록/최초 사용) | 등록 시 | 확정 |
| `memberCount` | 배정 직원 수 | 파생 | 확정 |
| `isGlobalDefault` | 전사 기본 근무조 여부 — 정확히 1개 | 지정 | 확정 (`:180`, `codeFlags` `:283`) |
| `usageLog` | 활성/비활성 상태 변경 이력 | 이력 | 확정 |
| `pendingChange` | 삭제·비활성화 전자결재 승인대기 상태 | 상신 시 | 확정 (`:980`) |

> 시간 산정 로직(연장·심야·반차 구간 등)은 근무조 속성의 파생값이며, 산정 규칙 자체는 [근무조 정책](../policies/WORK_SHIFT_POLICY.md)에서 다룬다.

## 4. 다른 엔티티와의 관계

| 상대 엔티티 | 관계 | 필수/선택 | 방향 | 상태 |
|---|---|---|---|---|
| 부서 (DEPARTMENT) | 부서가 근무조를 **기본 근무조/사용 근무조로 참조**. 근무조 코드가 먼저 존재해야 부서 추가 가능(선행) | 필수 | 근무조 → 부서 | 확정 (`page-hr-employee.js:1360~1364`; 프로세스맵 §4.1) |
| 직원 (EMPLOYEE) | 직원에게 근무조 배정(`e.shift = code`). 배정되면 "사용 중" | 조건부 | 근무조 → 직원 | 확정 (`shiftInUse()` `:69`) |
| 근무스케줄 (WORK_SCHEDULE) | 월별 편성이 근무조를 배치 단위로 사용 | 필수 | 근무조 → 스케줄 | 확정 (프로세스맵 §4.1, `page-att-shift-batch.js:8`) |
| 근태기록 (ATTENDANCE_RECORD) | 배치된 근무조가 근태 정상/이상 판정 기준 | 필수 | 근무조 → 근태 | 확정 (프로세스맵 §4.1) |
| 급여·식권 | 근무조·근무일이 초과수당/식권 산정에 참조 | 조건부 | 근무조 → 급여/식권 | 확정 (프로세스맵 §4.6) |

**필수 참조(선행)** — 근무조는 다른 업무의 선행 마스터다. 부서 추가 시 기본 근무조 선택이 필수이므로, 근무조 마스터에 코드가 먼저 등록되어 있어야 한다.

## 5. 상태 · 유효기간

- 상태 축은 두 가지다. ① **사용 상태**(파생): `사용 중(inuse)` / `미사용(unused)` / `승인대기(pending)` — 사용 부서 수 기준. ② **활성 축**: `활성(active)` / `비활성` — 비활성은 배치 후보에서 제외(폐기).
- 상세 상태 코드·전이·조건은 [WORK_SHIFT 상태 문서](../states/WORK_SHIFT_STATE.md) 참조.
- **유효기간** — 근무조 자체에 종료일 필드는 확인되지 않음. `startedAt`(사용 시작일)만 존재. 비활성화로 사용을 종료하며 별도 만료일은 두지 않는다. (확정 — 종료일 필드 부재 / 만료 정책은 `미결정`)

## 6. 불변 규칙 (Invariants)

| 규칙 | 내용 | 상태 |
|---|---|---|
| 코드 불변 | `code`는 채번 후 변경하지 않는다(식별자). | 확정 |
| 기본 근무조 유일 | `isGlobalDefault=true`인 근무조는 전사에 정확히 1개. | 확정 (`codeFlags` `:283`) |
| 사용 중 기준 잠금 | 사용 이력(배치)이 있는 근무조는 출·퇴근·휴게 등 산정 기준을 수정할 수 없다. | 확정 (`codeHasHistory` `:264`, `canEdit` `:290`) |
| D/N 자동 결정 | 심야 겹침 여부로 주간/야간이 자동 결정되며 사용자가 임의 지정하지 않는다. | 확정 (`:93`) |

## 7. 생성 · 변경 · 중지 액션

| 액션 | 엔티티 영향 | 전자결재 | 이력 | 상태 |
|---|---|---|---|---|
| 추가(등록) | 신규 근무조 생성(활성·미사용) | **없음(자유)** | 미기록 | 확정 (`saveAdd()` `:952`, 주석 `:213`) |
| 수정 | 산정 기준·명칭·색상 변경 | 없음 | 수정 이력(`shiftHistory`)에 기록. 부서 연결 근무조는 적용 시작일·사유 필수 | 확정 (`editMeta` `:242`, `diffShift` `:363`) |
| 미사용(비활성화) | `active=false`(배치 후보 제외) | **대상** → 승인대기 후 반영 | `codeChangeLog` + `usageLog` | 확정 (`submitCodeChange('비활성')` `:977`, `approveCodeChange` `:987`) |
| 재개(활성화) | `active=true` | 정책상 대상(주석) / 코드상 즉시 반영 | `codeChangeLog` + `usageLog` | 추정·구현 확인 (`activateCode` `:1003`는 결재 없이 즉시 처리, 주석 `:213`은 결재 대상) |
| 삭제 | 목록에서 제거 | **대상** → 승인대기 후 반영 | `codeChangeLog` | 확정 (`submitCodeChange('삭제')` `:977`) |

## 8. 이력 정책

- **변경 이력**(`codeChangeLog`) — 추가/삭제/미사용/재개 중 **삭제·미사용·재개만 전자결재·이력 대상**. 추가는 자유·미기록. 항목: 변경일시·대상 코드·명칭·유형·사유·변경자·승인자(최신순). (확정 `:235`, `:939`)
- **수정 이력**(`shiftHistory`) — 산정 기준·명칭 수정 시 필드별 before/after·적용시작일·사유·수정자 기록. (확정 `:1158`)
- **사용 상태 이력**(`usageLog`) — 활성/비활성 전환 타임라인. (확정 `:190`)

## 9. 삭제 정책

| 조건 | 삭제 가능 여부 | 상태 |
|---|---|---|
| 사용 부서 있음(`inUse`) | 불가 — 부서에서 먼저 해제 | 확정 (`canDelete` `:290`) |
| 사용 이력 있음(`everUsed`, 과거 배치 포함) | 불가 — 산정 기준 보존 | 확정 |
| 전사 기본 근무조(`isDefault`) | 불가 | 확정 |
| 승인대기(`pending`) | 불가(중복 상신 방지) | 확정 (`:1071`) |
| 위 모두 아님 + 활성 | 가능(전자결재 상신 → 승인 시 제거) | 확정 |

→ 실제 삭제는 소프트 여부(아카이브) 미확인. 목록에서 제거되고 변경 이력에는 남는다. 물리 삭제/보존 기간은 `미결정`.

---

## 10. 미결정 · 확인 사항

1. `미결정` — 근무조 만료일/유효기간 필드 없음. 사용 종료를 비활성으로만 표현하는지, 종료일 관리가 필요한지 담당자 확정 필요.
2. `추정·구현 확인` — 재개(활성화)가 전자결재 대상인지(주석) 즉시 반영인지(코드) 불일치. 확정 필요.
3. `미결정` — 삭제 시 물리 삭제 vs 아카이브, 이력 보존 기간.
4. `구현 갭` — 부서 연결 근무조 수정의 "적용 시작일" 예약(`promotePending`)이 실제 근태 소급/미소급에 미치는 영향은 백엔드 확인 대상.
