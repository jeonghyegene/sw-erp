## 1) Summary

**One-liner**: 고객 현장 입회(감리) 일정을 접수 목록 테이블과 주간 캘린더로 관리하며, 담당자·진행 일시 승인 시 고객에게 알림톡을 자동 발송하고, 감리유형 해당 공정의 생산 진행 시 자동 감리완료로 전환한다.

### Goals

- 감리대기 접수 건을 목록에서 확인하고 담당자·일정을 배정하여 승인 처리한다
- 승인된 감리 일정을 주간 캘린더 뷰로 시각화하여 일정 충돌을 방지한다
- 감리 승인 시 고객에게 일정 확정 알림톡을 자동 발송한다
- 날짜 범위·상태·검색어 필터로 감리 접수 목록을 빠르게 조회한다
- 감리유형 해당 공정의 생산 진행 시 감리완료 자동 전환으로 운영 부담을 최소화한다

### Scope

**In Scope**
- 감리 접수 목록 조회 (날짜 범위, 상태 필터, 텍스트 검색)
- 감리 상태 관리: 감리대기 → 감리승인 → 감리완료 (2단계 전이)
- 감리 일정 승인 (담당자 배정 + 진행 일시 지정)
- 승인 확정 시 고객 알림톡 자동 발송 + 발송 이력 기록
- 승인된 일정 수정 + 변경 알림톡 자동 재발송
- 노쇼 처리 (담당자 판단 → 일정 재조율 후 수동 재승인, 별도 노쇼 상태 없음)
- 주간 캘린더 뷰로 감리 일정 시각화 (감리유형별 색상 구분)
- GNB 감리대기 건수 뱃지 표시
- 인쇄/가공 공정 생산 진행 TRIGGER 수신 → 감리완료 자동 전환 (POL-MES-010)
- **인쇄/가공 공정 완료취소 롤백 TRIGGER 수신 → 감리승인 건 원점복귀(감리대기) + 감리자 알림 / 감리완료 건 Terminal 유지 + 감리자 알림만** (POL-MES-010, POL-MES-007, POL-PRI-002)
- **감리 대상 가공유형 동적 확장 대응**: 가공마스터의 감리대상여부 플래그에 따라 형압·박 등 확장 수용 (POL-MES-010)
- **감리자 시스템 알림 자동 발송**: 롤백 수신/감리 미승인 트리거/알림톡 실패 시 감리자에게 알림 (POL-PRI-002)
- 인쇄/가공 작업 카드에 감리 배지 표시 + 호버 툴팁 (depends_by 반영)

**Out of Scope**
- 감리 결과 입력 및 품질 판정 (합격/불합격 구분 없음, 완료 처리만 존재)
- 고객 직접 일정 입력 (홈페이지 영역)
- 재감리 프로세스 (별도 재감리 개념 없음)
- 감리 대상 작업 카드 배지·툴팁 렌더 로직은 WK-PRD-PRC / WK-PRD-FAB에서 담당

## 2) Decisions

| ID | 결정 사항 | 근거 |
|----|----------|------|
| DEC-001 | 감리 상태는 감리대기 → 감리승인 → 감리완료 2단계 전이로 한다. 감리거부 상태는 두지 않는다 | v3.0 확정 (이혁재). 감리는 생산 과정 동행이므로 거부 개념 불필요 |
| DEC-002 | 감리 승인 시 담당자 + 진행 일시 동시 필수 입력으로 한다 | POL-EXC-001 선행 조건 검증 |
| DEC-003 | 감리 승인 확정 시 고객에게 일정 확정 알림톡을 자동 발송한다 | POL-PRI-002 자동화 우선 |
| DEC-004 | 감리완료 트리거는 **PRC/FAB 공정 작업자의 "완료" 상태 변경 액션 시 노출되는 컨펌창**("감리완료처리 됩니다. 정말 완료하시겠습니까?")을 통해 작업자가 확정한 후 수신된다. 컨펌창은 작업 대상이 감리 대상 건(감리 배지 표시 대상)일 때만 노출되며, 확정 시 **공정 완료 처리 + 감리완료 전환이 동시에 이루어진다**. AUD 화면에는 수동 감리완료 버튼이 없다. 컨펌창 노출·검증 책임은 PRC/FAB Work MD가 담당하며, AUD는 확정 후 발행된 TRIGGER 수신·상태 전환만 담당한다 | v3.0 확정 + **v1.2.1 개정(사용자 확정 2026-04-22)**: 완전 자동 트리거 → 작업자 확인 컨펌창 확정 프로세스로 전환. 감리 건은 고객 현장 입회라는 물리적 의미가 있어 작업자가 명시적 확인 후 완료하는 것이 현장 운영 실태와 일치 |
| DEC-005 | 감리완료 자동 트리거는 감리상태가 감리승인일 때만 작동한다. 감리대기 상태에서 컨펌창 확정 시에는 **AUD 트리거를 적용하지 않고 감리대기를 유지**하며, 관리자에게 "감리 미승인 건의 해당 공정이 완료 처리되었습니다. 감리 처리를 확인해주세요." 알림을 발송한다. 작업자의 공정 완료 자체는 정상 처리된다 (공정과 감리 독립성 유지) | 감리 미승인 건은 일정 미확정 상태. v1.2.1: 컨펌창 프로세스에서도 동일 원칙 유지 |
| DEC-006 | 감리 일정 수정 시 고객에게 변경 알림톡을 자동 재발송한다 | POL-PRI-002 자동화 우선 |
| DEC-007 | 감리 결과 기록(합격/불합격)은 두지 않는다. 감리완료 처리만 존재한다 | 감리는 과정 동행 모니터링 성격 |
| DEC-008 | 노쇼는 별도 상태로 관리하지 않고 담당자가 일정 수정 + 재승인으로 처리한다 | POL-PRI-002 예외 수동 전환 |
| DEC-009 | 감리완료 건은 관리 버튼(감리승인/일정수정)을 표시하지 않는다 (Terminal 상태) | POL-APR-001 확정 후 복구 불가, POL-MES-007 역전이 차단 |
| DEC-010 | 진행상태(공정)와 감리상태는 별도 컬럼으로 분리하여 독립 관리한다 | 공정 생명주기와 감리 상태 독립성 |

### 공정 롤백 수신 (v1.1.0 신규)

| Rule ID | 규칙 내용 |
|---------|----------|
| LR-037 | PRC/FAB로부터 완료취소 롤백 TRIGGER 수신 시 해당 접수번호의 감리상태에 따라 3분기로 분리 처리한다: **(A) 감리승인** → 감리대기로 원점복귀, 담당자·진행일시 필드는 보존(이력 추적 목적), 캘린더 이벤트 카드 제거, 감리자 알림 발송. **(B) 감리완료** → Terminal 유지, 감리자 알림만 발송. **(C) 감리대기** → 상태 변경 없음, 이력만 기록. POL-MES-010, POL-MES-007을 따른다 |

| LR-042 | 감리 대상 가공유형은 WK-PRD-FAB가 가공마스터의 감리대상여부 플래그 기준으로 TRIGGER 발행 여부를 판단하며, AUD는 수신한 TRIGGER를 무조건 처리한다(감리유형 필드에 맞춰 접수건 식별). **마스터 설정 UI는 FAB 도메인 소재** — AUD에서는 책임 없음 (v1.2.2 사용자 확정 OI-052/도메인 OI-011). POL-MES-010, FAB DEC-28을 따른다 |
