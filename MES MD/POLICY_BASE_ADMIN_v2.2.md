## 2. 상태 전이 정책

### 2.1 공정 흐름

```
인쇄(CTP/디지털/옵셋) → 가공(오시/접지/코팅/제본) → 포장 → 출고
```

각 공정은 독립적인 상태 머신을 가진다. 선행 공정 완료가 후행 공정 진입의 전제 조건이다.

### 2.2 인쇄 작업 상태 전이

| 상태 | 코드 | 설명 |
|------|------|------|
| 대기 | waiting | 기기에 배정되어 출력 대기 중 |
| 진행 | active | 출력/CTP 작업 진행 중 |
| 완료 | completed | 인쇄 작업 완료 |
| 중지 | stopped | 작업 일시 중지 (기기 고장/자재 부족) |
| 재작업대기 | rework_waiting | 품질 불량으로 재인쇄 필요 |

**전이 매트릭스:**

| From → To | 조건 | 관련 정책 |
|-----------|------|----------|
| 대기 → 진행 | 스캔 시작 | POL-MES-004 |
| 진행 → 완료 | 작업 완료 처리 | — |
| 진행 → 중지 | 중지 사유 기록 | — |
| 중지 → 대기 | 중지 해제, 큐 재진입 | — |
| 완료 → 대기 | 완료취소 | POL-MES-006 |
| 완료 → 재작업대기 | 품질 이슈 발생 | — |
| 재작업대기 → 대기 | 재작업 배정 | — |

### 2.7 역전이 정책

**원칙**: 상태 전이는 단방향 강제. 아래 예외를 제외한 모든 역전이는 서버 검증으로 차단.

**허용된 역전이 (3건):**

| 역전이 | 조건 | 복귀 상태 | 근거 |
|--------|------|----------|------|
| 가공 완료 → 배정대기 | 사유 4종 중 1개 필수 선택 | 배정대기(assigned) | POL-MES-007 |
| 포장완료 → 포장대기 | 문제 발견 시 담당자 권한 | 포장대기(pkg_waiting) | 프레임워크 참조 |
| 인쇄 완료 → 대기 | 완료취소 (권한 차등) | 대기(waiting) | POL-MES-006 |

**보류(hold) 특수 규칙 (POL-MES-008, v2.2 개정):**
보류 해제 시 보류 직전 상태(가공대기 또는 가공중)로 복귀한다. 직전 상태 보존으로 이력 추적과 책임소재 명확성을 확보한다. v2.1 이전 "배정대기 고정 전환(큐 재진입)" 규칙은 폐기.

## 3. 권한 모델

### 3.1 MES 권한코드 (POL-MES-002)

| 권한코드 | 대상 | 화면접근 | F1(금전) | F2(개인정보) | F3(시스템) |
|----------|------|----------|----------|-------------|-----------|
| P_MES_WORKER | 현장 작업자 | 담당 기기 화면만 ON | OFF | OFF | OFF |
| P_MES_ADMIN | MES 관리자 | 전체 MES 화면 ON | OFF | OFF | 마스터데이터관리 ON, 수동상태보정 ON |

### 3.2 조장 역할

별도 권한코드 없음 (P_MES_WORKER와 동일). 차이점은 화면 유형만 통합형(여러 기기 작업 목록을 한 화면에서 조회). 기능 권한은 작업자와 동일.

### 3.3 화면별 접근 권한 매트릭스

| 화면 | 작업자 (P_MES_WORKER) | 관리자 (P_MES_ADMIN) |
|------|:---------------------:|:--------------------:|
| 로그인 (전체) | O | O |
| 홈 대시보드 | O (담당 기기만) | O (기기 영역 미표시) |
| 생산기기 목록 | O | O |
| 인쇄기기 상세 | O (담당 기기) | O (전체) |
| 가공기기 상세 | O (담당 기기) | O (전체) |
| 포장 상세 | O (담당 기기) | O (전체) |
| 모니터링 (전체) | — | O |
| 감리일정 관리 | — | O |
| 작업 상세 팝업 | O | O |
| 자재신청 팝업 | O | O |

### 3.4 기기PC 비밀번호 면제 (POL-MES-003)

기기 옆 전용PC에서 비밀번호 없이 담당자 카드 클릭 로그인 허용.

| 보완 조치 | 내용 |
|----------|------|
| 물리적 접근 통제 | 기기 위치 제한 |
| 감사 로그 | WHO/WHEN 전수 기록 |
| 적용 범위 | 기기PC 환경에서만 (관리자 웹은 일반 로그인 필수) |

### 3.5 비관적 Lock

MES 작업 데이터는 1기기 1작업자 전담이므로 동시 수정 충돌 가능성이 낮다. 비관적 Lock은 관리자의 마스터 데이터 편집에만 적용한다.

| 대상 | Lock 적용 |
|------|----------|
| 작업 데이터 (인쇄/가공/포장) | 미적용 |
| 감리 승인 모달 | 미적용 |
| 마스터 데이터 (장비/사용자/코드) | 적용 (프레임워크 참조: POL-AUTH-002) |

### 3.6 기기 상태 변경 권한

작업자 포함 전원이 기기 상태(가동중/종료/고장) 변경 가능. 본인 담당 기기 외에도 변경 허용.

## 5. 승인 체계

### 5.1 승인 면제 원칙

MES는 현장 생산 즉시성을 우선한다. 대부분의 MES 행위는 승인 없이 즉시 처리된다.

| 행위 | 승인 | 통제 방식 |
|------|------|----------|
| 작업 시작 (스캔) | 불필요 | POL-MES-004 (스캔 확인 1회) |
| 작업 완료 | 불필요 | 완료 처리 즉시 반영 |
| 작업 상태 되돌리기 | 불필요 | 사유 필수 기록 (POL-MES-007) |
| 보류/보류 해제 | 불필요 | 사유 기록 + 이력 보존 |
| 기기배정 (규칙 매칭 시) | 불필요 | 자동 배정 |
| CTP 추가사용 수정 | 불필요 | 즉시 반영 + 이력 기록 |
| 감리 승인/거부 | 불필요 | 감리 담당자 직접 처리 |

### 5.2 승인 필요 항목

| 행위 | 승인 단계 | 승인자 |
|------|----------|--------|
| 자재 신청 | 승인 필요 | 창고 담당자 (프레임워크 참조: POL-APR-003) |

### 5.3 완료취소 — 권한 차등 통제 (POL-MES-006)

## 6. 예외 처리

### 6.1 표준 Fallback 체인

MES 내 모든 자동화 처리 실패 시:

```
1단계: 자동 처리 시도
    ↓ 실패
2단계: 자동 재시도 (최대 3회, 간격: 즉시 → 30초 → 60초)
    ↓ 3회 모두 실패
3단계: 관리자 알림 발송 (시스템 내 알림)
    ↓ 관리자 확인
4단계: 수동 처리 전환
```

**예외**: 프린터/라벨러 출력 실패는 자동 재시도 없이 수동 재출력 버튼만 제공. 3회 연속 실패 시 관리자 알림.

### 6.2 선행 조건 검증 — 차단 우선

| 상황 | 차단 조건 | 안내 방식 |
|------|----------|----------|
| 포장 스캔 시 | 해당 접수의 가공 완료 여부 확인 | 미완료 시 "선행 공정 미완료" 안내 + 스캔 차단 |
| 인쇄 완료취소 시 | 후행 공정(가공) 진입 여부 확인 | 가공 진입 시 "가공 진행 중 — 완료취소 불가" 안내 |
| 고장 기기에서 작업 시작 | 기기 상태 = 고장/점검중/오프라인 | 작업 시작 차단 + "기기 점검 중" 안내 |
| 가공 스캔 시 | 인쇄 완료 여부 확인 | 미완료 시 스캔 차단 |
| 포장완료 → 출고대기 전환 시 | 감리 완료 여부 확인 (감리 대상인 경우) | 감리 미완료 시 전환 차단 |

**예외**: 감리는 공정 선행 검증 미적용 (담당자 수동 판단). 과정 동행 특성상 유동적 일정 반영.

### 6.3 친화적 안내 원칙

| 원칙 | 내용 |
|------|------|
| 기술 오류 미노출 | 내부 에러 코드·스택 트레이스 화면 미노출 |
| 한국어 안내 메시지 | 모든 오류/안내 메시지를 한국어로 표시 |
| 복구 불가 처리 전 확인 | 2단계 확인 팝업 적용 |
| 조치 가능한 안내 | "~할 수 없습니다" 대신 "~한 후 다시 시도해 주세요" |

### 6.4 외부 연동 실패 처리

**GW 인증 장애 시 오프라인 운영:**
GW 인증 API 3회 연속 실패 → 마지막 인증 세션 유지 오프라인 모드 → 작업 로그 로컬 임시 저장 → 복구 후 서버 일괄 전송 → 누락 데이터 관리자 확인.

**WebSocket 실패 시:**
WebSocket 끊김 → HTTP 폴링 자동 전환 (폴링 주기: 관리자 설정 가능) → HTTP도 실패 시 관리자 알림 + 수동 새로고침 안내 → 복구 시 WebSocket 자동 재연결.

**카카오 알림톡 실패 (감리):**
3회 재시도(즉시→30초→60초) → 담당자(관리자) 시스템 내 알림 → SMS Fallback 검토(운영 판단).

### 6.5 2단계 확인 팝업

복구 불가 상태 진입 전 반드시 2단계 확인 팝업 표시.

| 단계 | 내용 |
|------|------|
| 1단계 | 처리 내용 요약 + "진행하시겠습니까?" |
| 2단계 | "이 처리는 되돌릴 수 없습니다. 계속하시겠습니까?" (강조 문구) |

적용 대상: 출고 완료 처리, 관리자 수동상태보정.
